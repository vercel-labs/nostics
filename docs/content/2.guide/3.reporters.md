## What a reporter is

A reporter is a function called every time a diagnostic is raised.

```ts
type DiagnosticReporter<Opts extends object = object>
  = (diagnostic: Diagnostic, options: Opts) => void
```

You wire them at definition time:

```ts
defineDiagnostics({
  reporters: [reporterLog, devReporter, createFileReporter()],
  codes: { /* ... */ },
})
```

Every reporter fires on every call, in order. They can't stop one another. Errors thrown by a reporter propagate up to the caller, so reporters are expected to swallow their own failures (the built-in network reporter does this).

## Built-ins

| Reporter                       | Import path                | Default behaviour                                                                |
| ------------------------------ | -------------------------- | -------------------------------------------------------------------------------- |
| `reporterLog`                  | `nostics`                  | `console.warn(formatDiagnostic(d))`. Method configurable (see below).            |
| `reporterError`                | `nostics`                  | `console.error(formatDiagnostic(d))`.                                            |
| `createFileReporter(options?)` | `nostics/reporters/node`   | Appends each diagnostic as NDJSON to a local file (default `.nostics.log`).      |
| `createFetchReporter(url)`     | `nostics/reporters/fetch`  | POSTs the diagnostic JSON to `url`. Fetch failures are swallowed.                |
| `devReporter`                  | `nostics/reporters/dev`    | Sends the diagnostic over `import.meta.hot.send('nostics:report', ...)` to Vite. |

### `reporterLog`

Routes to `console[method]`. The method is opt-in via the second argument; default is `'warn'`.

```ts
import { reporterLog } from 'nostics'

defineDiagnostics({
  reporters: [reporterLog],
  codes: { /* ... */ },
})

// At the call site, pass method as the reporter options:
diagnostics.APP_E001(undefined, { method: 'error' })
```

If `method` is the only field you care about and you always want one value, pin it with a thin wrapper:

```ts
const reporterAlwaysError: DiagnosticReporter = d => reporterLog(d, { method: 'error' })
```

### `reporterError`

Convenience for `console.error`. No options.

```ts
import { reporterError } from 'nostics'

defineDiagnostics({
  reporters: [reporterError],
  codes: { /* ... */ },
})
```

### `createFileReporter`

Appends each diagnostic to a local file as a single NDJSON line. Useful for build tools, test runners, and dev servers.

```ts
import { defineDiagnostics } from 'nostics'
import { createFileReporter } from 'nostics/reporters/node'

defineDiagnostics({
  reporters: [
    createFileReporter({
      logFile: '.app-diagnostics.log',
      excludeStackFrames: [/\/node_modules\//, /\(node:/],
    }),
  ],
  codes: { /* ... */ },
})
```

| Option               | Type                | Default          | Description                                                                              |
| -------------------- | ------------------- | ---------------- | ---------------------------------------------------------------------------------------- |
| `logFile`            | `string`            | `'.nostics.log'` | Path to the output file. Relative to the process cwd.                                    |
| `excludeStackFrames` | `readonly RegExp[]` | `undefined`      | Frames matching any pattern are removed from `stack` before serialization.               |

Tail the file in another terminal while you develop:

```bash
tail -f .nostics.log | jq .
```

### `createFetchReporter`

POSTs each diagnostic as JSON to a URL. Fire-and-forget; fetch failures are swallowed so reporting never throws into user code.

```ts
import { createFetchReporter } from 'nostics/reporters/fetch'

defineDiagnostics({
  reporters: [createFetchReporter('https://telemetry.example.com/diagnostics')],
  codes: { /* ... */ },
})
```

The request body is `JSON.stringify(diagnostic)`, which calls `Diagnostic.toJSON()`. The shape includes `name`, `why`, `fix`, `docs`, `sources`, `cause`, and `stack`.

### `devReporter`

Designed for browser code running under Vite dev. Sends the diagnostic over `import.meta.hot.send('nostics:report', diagnostic.toJSON())`. On the server, the [Vite dev collector](/guide/dev-collector) listens and appends to a log file.

```ts
import { defineDiagnostics, reporterLog } from 'nostics'
import { devReporter } from 'nostics/reporters/dev'

export const diagnostics = defineDiagnostics({
  reporters: [reporterLog, devReporter],
  codes: { /* ... */ },
})
```

`reporterLog` still prints to the browser console. `devReporter` runs alongside it, not as a replacement. Outside Vite, `devReporter` warns once and does nothing.

See the [dev collector guide](/guide/dev-collector) for the server-side setup.

## Writing a custom reporter

A reporter is any function matching the signature. No registration, no lifecycle, no base class.

```ts
import type { DiagnosticReporter } from 'nostics'

const sentryReporter: DiagnosticReporter = (diagnostic) => {
  sentry.captureMessage(diagnostic.message, {
    tags: { code: diagnostic.name },
    extra: { fix: diagnostic.fix, docs: diagnostic.docs },
  })
}
```

Wire it like any built-in:

```ts
defineDiagnostics({
  reporters: [reporterLog, sentryReporter],
  codes: { /* ... */ },
})
```

## Required options

Reporters can declare an options shape via a generic. `defineDiagnostics` intersects every reporter's option shape into a single object passed at the call site. If any field is required, the call site signature is widened to require it.

```ts
import type { DiagnosticReporter } from 'nostics'

const audited: DiagnosticReporter<{ priority: number }> = (d, opts) => {
  audit.record({ code: d.name, priority: opts.priority })
}

const diagnostics = defineDiagnostics({
  reporters: [audited],
  codes: { APP_E001: { why: 'failed' } },
})

// The second argument is required and type-checked:
diagnostics.APP_E001(undefined, { priority: 1 })
```

If two reporters declare options, the merged shape is the intersection:

```ts
const a: DiagnosticReporter<{ priority: number }> = () => {}
const b: DiagnosticReporter<{ region: string }> = () => {}

const diagnostics = defineDiagnostics({
  reporters: [a, b],
  codes: { APP_E001: { why: 'failed' } },
})

diagnostics.APP_E001(undefined, { priority: 1, region: 'eu-west' })
```

Use this when a reporter genuinely needs per-call context (priority, severity escalation, request ID). Don't use it to plumb global config; create the reporter with the config baked in.

## Tips

- Order matters. Reporters run in declaration order. Put cheap ones first; put network ones last.
- Reporters should not throw. If yours can fail, wrap the body in `try/catch` and log the failure to `console.error`.
- For dev-time vs production setups, compose the array conditionally:

  ```ts
  reporters: [
    reporterLog,
    ...(import.meta.env.DEV ? [devReporter] : [createFetchReporter('/log')]),
  ]
  ```

- For library authors: do not require options at the call site. Required options force every consumer of your library to know about them. Build reporters with config baked in.
