## The shape

`defineDiagnostics` takes an options object and returns one callable handle per code. There is no `new`, no proxy. The returned object is typed.

```ts
import { defineDiagnostics, reporterLog } from 'nostics'

const diagnostics = defineDiagnostics({
  docsBase: 'https://example.com/errors',
  reporters: [reporterLog],
  codes: {
    APP_E001: {
      why: 'Could not load configuration file.',
      fix: 'Check that `app.config.ts` exists and exports a default object.',
    },
  },
})
```

`diagnostics.APP_E001` is a function. Calling it returns a `Diagnostic`. The reporters fire on every call.

## Static vs parameterised fields

Both `why` and `fix` accept either a string or a function. The function form interpolates a `params` object passed at the call site.

```ts
defineDiagnostics({
  codes: {
    MATH_E001: {
      why: 'Division by zero',
      fix: 'Ensure the denominator is not zero',
    },
    MATH_W001: {
      why: (p: { n: number }) => `Negative input ${p.n} for factorial`,
      fix: 'Ensure n is a non-negative integer',
    },
    MATH_W002: {
      why: 'Large input may overflow',
      fix: (p: { limit: number }) => `Avoid calling factorial with n > ${p.limit}`,
    },
  },
})
```

At the call site:

```ts
diagnostics.MATH_E001() // no params required
diagnostics.MATH_W001({ n: -3 }) // { n } required by `why`
diagnostics.MATH_W002({ limit: 170 }) // { limit } required by `fix`
```

## Parameter inference

Parameters from every templated field on a code are intersected. If `why` needs `{ src }` and `fix` needs `{ date }`, the call site requires `{ src, date }`.

```ts
const diagnostics = defineDiagnostics({
  codes: {
    NUXT_B5001: {
      why: (p: { configPath: string }) => `Missing compatibilityDate in ${p.configPath}.`,
      fix: (p: { date: string }) => `Add \`compatibilityDate: '${p.date}'\` to your config.`,
    },
  },
})

diagnostics.NUXT_B5001({
  configPath: 'nuxt.config.ts',
  date: '2024-04-03',
})
```

TypeScript surfaces missing keys at the call site, not in the definition.

## Runtime-only call-site fields

Two fields are only meaningful at the call site, not in the definition: `cause` and `sources`. Pass them in the same object as your params.

```ts
try {
  await loadConfig()
}
catch (cause) {
  throw diagnostics.NUXT_B5001({
    configPath: 'nuxt.config.ts',
    date: '2024-04-03',
    cause,
    sources: ['nuxt.config.ts:14:3'],
  })
}
```

- `cause` is preserved on the `Diagnostic` and on the underlying `Error`. Use it when re-raising.
- `sources` is an array of `file:line:column` strings. Useful for compilers and other tools where the JS stack doesn't reflect the user's source.

## Docs URLs

The `docsBase` option controls the `docs` URL on every diagnostic.

### String form

```ts
defineDiagnostics({
  docsBase: 'https://example.com/errors',
  codes: { APP_E001: { why: '...' } },
})
// â†’ diagnostics.APP_E001().docs === 'https://example.com/errors/app_e001'
```

The code is lowercased and appended. Good when your URL structure is regular.

### Function form

```ts
defineDiagnostics({
  docsBase: code => `https://nuxt.com/e/${code.replace('NUXT_', '').toLowerCase()}`,
  codes: { NUXT_B2011: { why: '...' } },
})
// â†’ diagnostics.NUXT_B2011().docs === 'https://nuxt.com/e/b2011'
```

Use the function form when you need to transform the code, strip a prefix, or use a different URL structure per category.

### Per-code overrides

A code's `docs` field overrides `docsBase` for that one code:

```ts
defineDiagnostics({
  codes: {
    APP_E001: {
      why: '...',
      docs: 'https://example.com/legacy/app_e001', // explicit URL
    },
    APP_W900: {
      why: '...',
      docs: false, // no `see:` line, no `docs` property on the Diagnostic
    },
  },
})
```

`docs: false` opts a code out of the docs URL entirely, even when `docsBase` is set.

## Naming conventions

Codes are permanent identifiers. Once published, never reuse, never rename, never repurpose.

A workable scheme:

```
<PREFIX>_<CATEGORY><NNNN>
```

- **PREFIX**: project or domain in uppercase. `NUXT`, `MATH`, `I18N`.
- **CATEGORY**: a single letter you assign meaning to. Common: `E` (error), `W` (warning), `D` (deprecation), `B` (build), `R` (runtime), `C` (config), `I` (info).
- **NNNN**: zero-padded sequence within the category.

```
MATH_E001
MATH_W001
NUXT_B2011
NUXT_D0042
```

Pick the scheme once. Document it next to your `defineDiagnostics` call. New codes always get the next free number.

## Organizing across files

For a small library, one file with one `defineDiagnostics` call is fine. For larger projects, split by domain:

```
src/
  diagnostics/
    build.ts       # MATH_B-series codes
    runtime.ts     # MATH_R-series codes
    config.ts      # MATH_C-series codes
    index.ts       # re-exports the handles
```

Each file calls `defineDiagnostics` with the same `docsBase` but a different code range. Consumers import the relevant handle directly.

```ts [src/diagnostics/index.ts]
export { diagnostics as buildDiagnostics } from './build'
export { diagnostics as configDiagnostics } from './config'
export { diagnostics as runtimeDiagnostics } from './runtime'
```

If you want a single object that exposes everything, merge after the fact:

```ts
export const diagnostics = {
  ...buildDiagnostics,
  ...runtimeDiagnostics,
  ...configDiagnostics,
}
```

::callout{icon="i-lucide-triangle-alert" color="warning"}
For build-time stripping to track these handles across files, prefer direct re-exports of the `defineDiagnostics` result. See [tree-shaking limits](/guide/production-builds#cross-file-tracking).
::
