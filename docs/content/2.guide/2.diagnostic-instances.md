## What you get back

Every handle returns a `Diagnostic` instance. The class extends `Error`, so anything that accepts an `Error` accepts a `Diagnostic`.

```ts
class Diagnostic extends Error {
  name: string // the code, e.g. 'MATH_E001'
  message: string // the already-interpolated `why`
  get why(): string // alias for `message`
  fix?: string // resolved from the definition
  docs?: string // resolved from docsBase / per-code docs
  sources?: string[] // call-site `sources`
  cause?: unknown // call-site `cause`
  toJSON(): object // structured payload
}
```

`name` is the code. `message` is the resolved `why`. `why` is an alias for `message` so you can read either.

## Report or throw

Calling a handle always builds the diagnostic and runs every reporter. What happens next is up to you.

```ts
// Report only. Code keeps running.
diagnostics.MATH_W001({ n })

// Throw. Code stops.
throw diagnostics.MATH_E001()
```

`throw` is just JavaScript. There is no separate "throw" method. The same handle returns a `Diagnostic` you can decide to raise or not.

Common patterns:

```ts
// Warn during normal operation.
if (n < 0) {
  diagnostics.MATH_W001({ n })
  return 1
}

// Hard error on invariant break.
if (b === 0) {
  throw diagnostics.MATH_E001()
}

// Re-raise from a caught error, preserving the cause.
try {
  await loadConfig()
}
catch (cause) {
  throw diagnostics.APP_E001({ cause })
}
```

## Catching diagnostics

`instanceof Diagnostic` narrows the catch to a typed surface.

```ts
import { Diagnostic } from 'nostics'

try {
  divide(1, 0)
}
catch (err) {
  if (err instanceof Diagnostic) {
    console.log(err.name) // 'MATH_E001'
    console.log(err.message) // 'Division by zero'
    console.log(err.fix) // 'Ensure the denominator is not zero'
    console.log(err.docs) // 'https://example.com/docs/diagnostics/math_e001'
  }
}
```

To dispatch on a specific code, switch on `name`:

```ts
try {
  doWork()
}
catch (err) {
  if (err instanceof Diagnostic) {
    switch (err.name) {
      case 'MATH_E001':
        // ...
        break
      case 'MATH_W001':
        // ...
        break
    }
  }
}
```

## Serializing

`Diagnostic.toJSON()` returns a plain object suitable for sending across process or network boundaries. `JSON.stringify(diagnostic)` calls it automatically.

```ts
const d = diagnostics.MATH_W001({ n: -3 })
JSON.stringify(d)
```

```json
{
  "name": "MATH_W001",
  "why": "Negative input -3 for factorial",
  "fix": "Ensure n is a non-negative integer",
  "docs": "https://example.com/docs/diagnostics/math_w001",
  "sources": null,
  "cause": null,
  "stack": "Diagnostic: Negative input -3 for factorial\n    at ..."
}
```

`sources` and `cause` serialize to `null` when omitted. `stack` is included so you can correlate diagnostics raised in one process with their origin in another. Reporters that ship diagnostics to a remote service rely on this.

## Stack traces

By default, the captured stack starts at the `Diagnostic` constructor call. `defineDiagnostics` passes its internal action method to `Error.captureStackTrace`, so the top frame is the user code that invoked the handle, not nostics internals.

On engines without `Error.captureStackTrace` (everything non-V8), nostics falls back to the default `Error` stack, which includes a couple of internal frames but is still usable.

If you want to filter out specific frames, the [file reporter](/guide/reporters#createfilereporter) accepts an `excludeStackFrames` option.

## Identity tips

- `diagnostic.name` is the code. Use it for dispatch.
- `diagnostic.message` is the human text. Use it for display.
- `diagnostic.why` is the same string as `diagnostic.message`. Pick one in your codebase.
- Don't read or write fields other than the documented ones. The class is intentionally small.
