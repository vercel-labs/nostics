## The problem

Browser diagnostics live in the browser console. A terminal-bound user (or an AI agent driving the project) can't see them. By the time the developer notices and copies the message, half the context is gone.

The dev collector closes the loop. Browser code reports a diagnostic; the Vite dev server forwards it into a local log file you can `tail`, grep, or feed to whatever agent is watching.

## How it works

Two pieces, on opposite sides of the dev channel:

```
┌──────────────── browser ────────────────┐    ┌───────── Vite dev server ─────────┐
│                                         │    │                                   │
│  diagnostics.MATH_E001()                │    │                                   │
│       │                                 │    │                                   │
│       ▼                                 │    │                                   │
│  devReporter (from nostics/reporters/dev) ───WS───▶ nosticsCollector             │
│       │                                 │    │       │                           │
│       ▼ also runs                       │    │       ▼                           │
│  reporterLog (console.warn)             │    │  appendFileSync('.nostics.log')   │
│                                         │    │                                   │
└─────────────────────────────────────────┘    └───────────────────────────────────┘
```

- `devReporter` is a client reporter you add to your `defineDiagnostics` call. It serializes each diagnostic to JSON and sends it over Vite's HMR WebSocket with `import.meta.hot.send('nostics:report', payload)`.
- `nosticsCollector` is a Vite plugin that listens for the same event on `server.ws` and appends every diagnostic to a local file via the [file reporter](/guide/reporters#createfilereporter).

`reporterLog` (or whatever else you wire) keeps running on the client. The collector is additive, not a replacement.

## Vite only

`nosticsCollector` relies on Vite's `configureServer` and `server.ws.on(...)` APIs. There are no equivalents in Webpack, Rollup, Rolldown, Rspack, esbuild, or Farm. Those adapters exist on the plugin object (because unplugin requires them) but they are no-ops. Use Vite for the dev experience; the build-time stripping plugin works everywhere.

## Setup

### 1. Wire `devReporter` in your client code

```ts [src/diagnostics.ts]
import { defineDiagnostics, reporterLog } from 'nostics'
import { devReporter } from 'nostics/reporters/dev'

export const diagnostics = defineDiagnostics({
  reporters: [reporterLog, devReporter],
  codes: { /* ... */ },
})
```

Order matters only for output ordering. Both reporters run on every call.

### 2. Add the collector to your Vite config

```ts [vite.config.ts]
import { nosticsCollector } from 'nostics/unplugin/dev-server-collector'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [nosticsCollector.vite()],
})
```

When the dev server starts, you'll see:

```
📋 nostics ··· saving logs to .nostics.log
```

### 3. Tail the log

```bash [Terminal]
tail -f .nostics.log | jq .
```

Every diagnostic raised in the browser appears as one JSON line, with `name`, `why`, `fix`, `docs`, `sources`, `cause`, and `stack`.

## Options

```ts
nosticsCollector.vite({
  logFile: '.app-diagnostics.log',
  debug: true,
  excludeStackFrames: [/\/node_modules\//, /\(node:/],
})
```

| Option               | Type                | Default                | Description                                                                                |
| -------------------- | ------------------- | ---------------------- | ------------------------------------------------------------------------------------------ |
| `logFile`            | `string`            | `'.nostics.log'`       | Path to the output file, resolved against the Vite project root.                           |
| `debug`              | `boolean`           | `!!process.env.DEBUG`  | When true, logs `[nostics]` lines to the Vite server console on every event.               |
| `excludeStackFrames` | `readonly RegExp[]` | `undefined`            | Frames matching any pattern are stripped from each diagnostic's stack before it is written. |

`excludeStackFrames` is forwarded to `createFileReporter`. The `Error: ...` header line is always preserved; only frame lines are filtered.

## When the collector is the wrong tool

- **Production telemetry.** The collector is dev-only. For production logging, use [`createFetchReporter`](/guide/reporters#createfetchreporter) or your own reporter.
- **Bundlers other than Vite.** No equivalent. If you need cross-bundler dev capture, ship your own WebSocket bridge.
- **Server-side code already running under Node.** Wire `createFileReporter` directly. The dev collector is purely a browser-to-server bridge.

## Adding the log file to `.gitignore`

```gitignore
.nostics.log
```

Or whatever path you configured. It's append-only and grows for as long as the dev server is up.
