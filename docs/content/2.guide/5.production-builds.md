## The problem

Diagnostics carry weight your production users don't pay for:

- Template strings with parameter interpolation.
- The `Diagnostic` class and its `toJSON()`.
- The reporters wired at definition time.
- Whatever your `fix` and `why` strings spell out.

A library that defines a hundred codes can easily ship a few kilobytes of dead text to every user. nostics gives you a build-time plugin that removes it.

## What the plugin does

`nostics/unplugin/strip-transform` exports an [unplugin](https://github.com/unjs/unplugin) you wire into your bundler. It performs two transforms on each source file that imports from `nostics`:

1. **Marks `defineDiagnostics()` calls as `/*#__PURE__*/`.** Bundlers treat the call as side-effect-free. If nothing in the production bundle reads the resulting object, the entire call is dropped.
2. **Guards diagnostic call sites with `NODE_ENV`.** Each expression statement that uses a diagnostic handle is rewritten to:

   ```ts
   process.env.NODE_ENV !== 'production' && diagnostics.MATH_W001({ n })
   ```

   In a production build with `NODE_ENV` defined (or via Vite's `import.meta.env.PROD` equivalent), the guard evaluates to `false` at build time, the call is dropped, and the handle becomes unreferenced. The `/*#__PURE__*/` on the original `defineDiagnostics` call then makes the whole catalog disappear.

`throw diagnostics.MATH_E001()` statements stay. The transform only rewrites bare expression statements; it doesn't touch `throw`, `return`, or assignments. Throws are part of your library's contract and should remain in production.

## Setup

Vite:

```ts [vite.config.ts]
import { nosticsStrip } from 'nostics/unplugin/strip-transform'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [nosticsStrip.vite()],
})
```

Rollup or Rolldown (e.g. inside `tsdown`):

```ts [tsdown.config.ts]
import { nosticsStrip } from 'nostics/unplugin/strip-transform'
import { defineConfig } from 'tsdown'

export default defineConfig({
  entry: ['src/index.ts'],
  plugins: [nosticsStrip.rolldown()],
})
```

The plugin is shipped via [unplugin](https://github.com/unjs/unplugin), so you can also use `.rollup()`, `.webpack()`, `.rspack()`, `.esbuild()`, or `.farm()`.

## What gets transformed

The plugin only touches `.js`, `.jsx`, `.ts`, `.tsx`, `.mjs`, and `.mts` files. It explicitly skips files under `node_modules/`.

Files are transformed when:

- They import any symbol from the configured package (default `nostics`), **or**
- They import (via a relative path) a variable that another tracked file derived from a nostics call.

If neither applies, the plugin returns the source unchanged.

## Cross-file tracking

When you split your diagnostics out of the file that raises them, the plugin walks one hop of relative imports to find the original `defineDiagnostics` call.

```ts [src/diagnostics.ts]
import { defineDiagnostics } from 'nostics'

export const diagnostics = defineDiagnostics({ codes: { /* ... */ } })
```

```ts [src/math.ts]
import { diagnostics } from './diagnostics'

export function divide(a: number, b: number): number {
  if (b === 0) {
    diagnostics.MATH_E001() // ŌåÉ guarded by NODE_ENV
  }
  return a / b
}
```

The plugin recognises that `diagnostics` was derived from `defineDiagnostics` in a relative neighbour, and wraps the call site appropriately.

### Limits

- **Relative imports only.** Imports that don't start with `.` are ignored.
- **One hop.** The plugin analyses the module you import directly. It does not chase barrel re-exports (`export { diagnostics } from './a'` re-exporting from another file is **not** followed).
- **Direct `defineDiagnostics()` calls.** Only `const x = defineDiagnostics(...)` and `export const x = defineDiagnostics(...)` are recognised. Wrapping in a factory (`export const x = makeDiagnostics()`) defeats tracking.

If you split diagnostics across files, keep the `defineDiagnostics` call adjacent to its export and import it directly from the consumer.

## Options

```ts
nosticsStrip.vite({
  packageName: 'my-fork', // detect imports from this package instead of 'nostics'
})
```

| Option        | Type     | Default     | Description                                                   |
| ------------- | -------- | ----------- | ------------------------------------------------------------- |
| `packageName` | `string` | `'nostics'` | Package name to detect imports from. Useful for forks or wrappers. |

## When the bundler ignores `NODE_ENV`

The guard relies on your bundler replacing `process.env.NODE_ENV` with a constant at build time. Vite, Rollup, Webpack, and esbuild all do this by default in production mode. If you bundle without that replacement (for example, running source directly in Node), the guard becomes a real runtime check, and diagnostics still run.

For Vite specifically: `import.meta.env.PROD` and `process.env.NODE_ENV` are both replaced by the env plugin in production builds. No extra config required.

## Verifying

After a production build, grep the output:

```bash
rg 'MATH_E001|MATH_W001' dist/
```

Throws should still appear (string literals inside `throw diagnostics.X()` survive). Bare expression statements should not. If you find unexpected references, the plugin probably couldn't track them across files; see [cross-file limits](#limits).
