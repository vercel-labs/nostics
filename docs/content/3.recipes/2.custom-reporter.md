A reporter is `(diagnostic: Diagnostic, options?: Opts) => void`. There is no registration step. Anything matching the signature works.

## A Sentry reporter

Pipe diagnostics into your error tracker with name and metadata preserved.

```ts
import type { DiagnosticReporter } from 'nostics'
import * as Sentry from '@sentry/browser'

const sentryReporter: DiagnosticReporter = (diagnostic) => {
  Sentry.captureMessage(diagnostic.message, {
    tags: { code: diagnostic.name },
    extra: {
      fix: diagnostic.fix,
      docs: diagnostic.docs,
      sources: diagnostic.sources,
    },
  })
}
```

Wire it like any built-in:

```ts
defineDiagnostics({
  reporters: [reporterLog, sentryReporter],
  codes: { /* ... */ },
})
```

The reporter never throws, so a Sentry outage can't take down the calling code.

## An async network reporter

Reporters return `void`. If you want to do async work, fire and forget. Swallow failures so reporting doesn't surface in user code.

```ts
import type { DiagnosticReporter } from 'nostics'

const remoteReporter: DiagnosticReporter = (diagnostic) => {
  fetch('https://telemetry.example.com/diagnostics', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(diagnostic),
  }).catch(() => {
    // network failures should not throw into user code
  })
}
```

This is essentially what `createFetchReporter` does. Use the built-in unless you need to add headers, sign the request, or batch.

## A reporter with required options

Reporters can declare a typed options shape. When you pass them to `defineDiagnostics`, the resulting call site signature requires the options.

```ts
import type { DiagnosticReporter } from 'nostics'

const audited: DiagnosticReporter<{ priority: number }> = (d, opts) => {
  audit.record({ code: d.name, priority: opts.priority })
}

const diagnostics = defineDiagnostics({
  reporters: [audited],
  codes: {
    APP_E001: { why: 'Could not load config' },
  },
})

// TypeScript requires the second argument:
diagnostics.APP_E001(undefined, { priority: 1 })
```

When two reporters declare options, the merged shape is the intersection:

```ts
const a: DiagnosticReporter<{ priority: number }> = () => {}
const b: DiagnosticReporter<{ region: string }> = () => {}

const diagnostics = defineDiagnostics({
  reporters: [a, b],
  codes: { APP_E001: { why: 'failed' } },
})

diagnostics.APP_E001(undefined, { priority: 1, region: 'eu-west' })
```

**Watch out**: required options become required for every call site of every code. If the option only matters sometimes, make it optional in the reporter signature and check it at runtime, or bake the value into the reporter at construction.

## A reporter built with config

For library authors, baking config into the reporter is usually nicer than requiring options at the call site. Consumers of your library shouldn't need to know about the auditing system.

```ts
function createAuditedReporter(opts: {
  endpoint: string
  priorityFor: (code: string) => number
}): DiagnosticReporter {
  return (d) => {
    fetch(opts.endpoint, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ code: d.name, priority: opts.priorityFor(d.name) }),
    }).catch(() => {})
  }
}

defineDiagnostics({
  reporters: [
    createAuditedReporter({
      endpoint: '/internal/audit',
      priorityFor: code => code.startsWith('APP_E') ? 1 : 3,
    }),
  ],
  codes: { /* ... */ },
})
```

This pattern keeps the call site clean and the wiring local to where the configuration lives.

## A test reporter

For tests, collect diagnostics into an array instead of side-effecting them.

```ts
import type { Diagnostic, DiagnosticReporter } from 'nostics'

function createCollector(): {
  reporter: DiagnosticReporter
  collected: Diagnostic[]
} {
  const collected: Diagnostic[] = []
  return {
    collected,
    reporter: d => collected.push(d),
  }
}

const { reporter, collected } = createCollector()

const diagnostics = defineDiagnostics({
  reporters: [reporter],
  codes: { APP_E001: { why: 'failed' } },
})

diagnostics.APP_E001()
expect(collected).toHaveLength(1)
expect(collected[0].name).toBe('APP_E001')
```

The same shape works for snapshot tests by formatting with `formatDiagnostic` or `jsonFormatter` before asserting.

## Combining reporters

There is no router. If you want different reporters for different code prefixes, gate inside a single reporter:

```ts
const router: DiagnosticReporter = (d, opts) => {
  if (d.name.startsWith('APP_E')) {
    sentryReporter(d, opts)
  }
  else if (d.name.startsWith('APP_D')) {
    deprecationReporter(d, opts)
  }
}
```

Keep the routing logic visible. Spreading it across multiple reporters that each filter by code makes the wiring harder to read.
