This recipe walks the full path from definition to dev-time capture, using the `demo-lib` + `playground` projects that ship in the nostics repo. The same shape applies to any library + app pair.

## The setup

Two projects in a workspace:

```
.
├── demo-lib/         # the library
│   ├── src/
│   │   ├── diagnostics.ts
│   │   ├── math.ts
│   │   └── index.ts
│   └── tsdown.config.ts
└── playground/       # the consumer (Vue + Vite)
    ├── src/App.vue
    └── vite.config.ts
```

The library publishes math utilities. The consumer is a Vue app with buttons that call them, including buttons that deliberately trigger each diagnostic.

## Step 1: define the catalog in the library

```ts [demo-lib/src/diagnostics.ts]
import { defineDiagnostics, reporterLog } from 'nostics'
import { devReporter } from 'nostics/reporters/dev'

export const diagnostics = defineDiagnostics({
  docsBase: code => `https://example.com/docs/diagnostics/${code.toLowerCase()}`,
  reporters: [reporterLog, devReporter],
  codes: {
    MATH_E001: {
      why: 'Division by zero',
      fix: 'Ensure the denominator is not zero',
    },
    MATH_W001: {
      why: (p: { n: number }) => `Negative input ${p.n} for factorial`,
      fix: 'Ensure n is a non-negative integer',
    },
    MATH_W002: {
      why: (p: { n: number }) => `Large input ${p.n} may overflow`,
      fix: 'Avoid calling factorial with n > 170',
    },
    MATH_D001: {
      why: 'sum() is deprecated',
      fix: 'Use add() instead',
    },
  },
})
```

Two reporters. `reporterLog` writes to the browser console for the user to see. `devReporter` ships a JSON copy over Vite's HMR channel for the dev collector to file.

## Step 2: raise diagnostics where it matters

```ts [demo-lib/src/math.ts]
import { diagnostics } from './diagnostics'

export function add(a: number, b: number): number {
  return a + b
}

export function divide(a: number, b: number): number {
  if (b === 0) {
    diagnostics.MATH_E001()
  }
  return a / b
}

export function factorial(n: number): number {
  if (n < 0) {
    diagnostics.MATH_W001({ n })
  }
  if (n > 170) {
    diagnostics.MATH_W002({ n })
  }
  if (n <= 1)
    return 1
  return n * factorial(n - 1)
}

export function sum(a: number, b: number): number {
  diagnostics.MATH_D001()
  return add(a, b)
}
```

Notice that none of these `throw`. The library reports and continues. The consumer decides whether to escalate.

`MATH_E001` raises on division by zero (the result is `Infinity`, which the library lets through). `MATH_W001` and `MATH_W002` warn on out-of-range factorial inputs. `MATH_D001` warns once per `sum()` call: it's a deprecation banner pointing at `add()`.

## Step 3: strip the catalog from the library's production build

The library is published as ESM via `tsdown`. The build config wires `nosticsStrip.rolldown()`:

```ts [demo-lib/tsdown.config.ts]
import { nosticsStrip } from 'nostics/unplugin/strip-transform'
import { defineConfig } from 'tsdown'

export default defineConfig({
  entry: ['src/index.ts'],
  sourcemap: true,
  exports: true,
  deps: {
    neverBundle: ['nostics'],
    onlyBundle: [],
  },
  plugins: [nosticsStrip.rolldown()],
})
```

Two things to call out:

- `nosticsStrip.rolldown()` rewrites every diagnostic call site with the `NODE_ENV` guard and marks `defineDiagnostics()` pure. In a production build, the entire catalog drops out.
- `neverBundle: ['nostics']` keeps `nostics` as an external runtime dependency. The library doesn't ship a copy of nostics inlined; consumers resolve it themselves.

In dev (where `NODE_ENV !== 'production'`), the guards are still emitted but evaluate to `true`. Diagnostics fire normally. The plugin is build-time only; it costs nothing at runtime.

## Step 4: wire the dev collector in the consumer

```ts [playground/vite.config.ts]
import vue from '@vitejs/plugin-vue'
import { nosticsCollector } from 'nostics/unplugin/dev-server-collector'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [
    vue(),
    nosticsCollector.vite(),
  ],
})
```

That's the whole consumer-side wire-up. Start the dev server:

```bash
pnpm --filter playground dev
```

Vite logs:

```
📋 nostics ··· saving logs to .nostics.log
```

## Step 5: trigger diagnostics in the UI

The Vue app exposes one section per math function, with regular inputs and "warn" buttons that deliberately trigger each code:

```vue [playground/src/App.vue]
<button class="warn" @click="runDivide(1, 0)">Divide by zero</button>
<button class="warn" @click="runFactorial(-5)">Negative (-5)</button>
<button class="warn" @click="runFactorial(200)">Overflow (200)</button>
```

Click any of them. Two things happen in parallel:

- The browser console shows the formatted diagnostic (because `reporterLog` is wired).
- `.nostics.log` gains one NDJSON line (because `devReporter` shipped it over HMR and `nosticsCollector` filed it).

Tail the log in another terminal:

```bash
tail -f playground/.nostics.log | jq .
```

```json
{
  "name": "MATH_W001",
  "why": "Negative input -5 for factorial",
  "fix": "Ensure n is a non-negative integer",
  "docs": "https://example.com/docs/diagnostics/math_w001",
  "sources": null,
  "cause": null,
  "stack": "Diagnostic: Negative input -5 for factorial\n    at factorial (..."
}
```

That's the loop. The library author owns the catalog. The consumer wires one Vite plugin and gets a stream of diagnostics they (or their agent) can actually read.

## Step 6: ship to production

When the consumer runs `vite build`, two things happen:

- `nosticsCollector` is a dev-only plugin (it only registers `configureServer`). It is a no-op at build time.
- The consumer imports `divide`, `factorial`, `sum` from `@posva/lib-demo`. That library was already built with `nosticsStrip`, so the diagnostic call sites in the published code are guarded with the `NODE_ENV` check. Vite replaces `process.env.NODE_ENV` with `'production'` at build time, the guards short-circuit, and the diagnostic catalog disappears from the bundle.

`throw diagnostics.X()` statements would survive, because the transform leaves throws alone. The demo library only uses bare expression statements, so its production bundle ships zero diagnostic code.

## Reusing this shape

The pattern works for any library/consumer split:

- **Library** wires `[reporterLog, devReporter]` and runs `nosticsStrip` in its build.
- **Consumer** wires `nosticsCollector.vite()` in dev. Nothing else.

If the consumer also wants a permanent off-server log, add a `createFetchReporter` to the library's reporters list (or, more cleanly, expose the reporter array as a library option so consumers can compose their own).
