Every diagnostic code carries a `docs` URL. Once you set `docsBase`, you're committing to a public page per code. This guide is about what to put on that page, how to host it, and how to keep it in sync with the code that emits the diagnostics.

## Why the registry matters

Three audiences land on `docs.example.com/errors/app_e001`:

1. **Developers** copying the URL from their terminal because the diagnostic told them to. They want a fix in 30 seconds.
2. **AI agents** (Claude, Copilot, and friends) fetching the page when a user pastes an error. They want unambiguous, structured content they can act on.
3. **Search engines** indexing the code. A developer searches `APP_E001` and lands here.

All three benefit from the same things: stable URLs, predictable structure, and content that's readable without JavaScript.

## Picking your URL shape

Use the `docsBase` option of `defineDiagnostics` to commit to a URL shape. String form when the structure is regular:

```ts
defineDiagnostics({
  docsBase: 'https://example.com/errors',
  codes: { APP_E001: { why: '...' } },
})
// → https://example.com/errors/app_e001
```

Function form when you need transformation:

```ts
defineDiagnostics({
  docsBase: code => `https://nuxt.com/e/${code.replace('NUXT_', '').toLowerCase()}`,
  codes: { NUXT_B2011: { why: '...' } },
})
// → https://nuxt.com/e/b2011
```

Whatever you pick, treat URLs as permanent. The terminal output that points at them goes out to logs, dashboards, agent traces, and users' shells. A 404 a year later is a poor experience.

## Page structure

Every page should hit the same sections in the same order. Predictability is the value here: a human can scan it, and an agent can extract content without ambiguity.

### Required sections

#### Title and code identifier

```markdown
# NUXT_B2011: Invalid plugin, src option is required

Code: `NUXT_B2011`
Level: error
```

Lead with the code and a short title. Include the code on its own line and a `Level:` field (`error`, `warning`, `deprecation`, `info`). Agents read both.

#### What this error means

```markdown
## What this error means

This error occurs when a plugin is registered via `addPlugin()` without
providing a valid `src` path. Nuxt requires every plugin to have a source
file so it can be resolved and included in the build.
```

Plain language. Assume no prior context. Describe what the system expected versus what it received. This is the section agents quote when explaining the error to users.

#### Why this happens

```markdown
## Why this happens

Common causes:

- Passing an object to `addPlugin()` without a `src` property
- Passing `undefined` or `null` as the plugin path
- A module constructing a plugin object dynamically with a missing `src`
  due to a conditional branch or typo
```

A bulleted list of concrete scenarios. Each bullet is one specific situation a developer might be in.

#### How to fix it

````markdown
## How to fix it

Ensure every call to `addPlugin()` includes a valid `src` path:

```ts
// Wrong
addPlugin({ name: 'my-plugin' })

// Correct
addPlugin({ src: resolve('./runtime/my-plugin'), name: 'my-plugin' })

// Also correct, pass a string directly
addPlugin(resolve('./runtime/my-plugin'))
```

If the plugin path is computed dynamically, verify the variable is defined
before passing it to `addPlugin()`.
````

This is the most important section. Show the wrong pattern and the corrected version. Examples should be copy-pasteable.

### Optional sections

#### Additional context

```markdown
## Additional context

- This validation was added in Nuxt 3.2.
- See the [Module Author Guide](https://nuxt.com/docs/guide/going-further/modules) when writing a Nuxt module.
- Related codes: [NUXT_B1001](./nuxt_b1001), [NUXT_B3005](./nuxt_b3005).
```

Link to related docs, changelog entries, or related codes.

#### Example terminal output

````markdown
## Example output

```
[NUXT_B2011] Invalid plugin `/plugins/bad.ts`. src option is required.
├▶ fix: Pass a string path or an object with a `src` property to `addPlugin()`.
├▶ sources: nuxt.config.ts:42:3
╰▶ see: https://nuxt.com/e/b2011
```
````

Show the user what they should expect to see in their terminal. Helps them confirm they're on the right page.

## Page template

```markdown
# {CODE}: {Short title}

Code: `{CODE}`
Level: {error|warning|deprecation|info}

## What this error means

{Plain-language explanation, 1 to 3 sentences.}

## Why this happens

{Bulleted list of concrete scenarios.}

## How to fix it

{Wrong vs corrected code examples.}

## Additional context

{Optional. Links to related docs, changelog, related codes.}

## Example output

{Optional. Terminal output showing the formatted diagnostic.}
```

Use this template for every code. Variation hurts both humans and agents.

## Deployment recommendations

- **Host the pages on a stable, public URL** matching `docsBase`. A static site (VitePress, Nuxt Content, Docus, MkDocs, ...) is enough.
- **Return proper HTTP status codes.** 200 for valid codes, 404 for unknown ones. Agents and crawlers tell missing codes from valid ones via status.
- **Avoid client-side-only rendering.** Agents that fetch the page may not run JavaScript. Keep the content in the initial HTML.
- **Add structured frontmatter or `<meta>` tags** with `code`, `level`, and `title` fields. Search engines and agents parse them.
- **Keep pages lightweight.** Heavy SPAs block content rendering for fetch-based agents.

## Keeping docs in sync with code

The catalog and the docs site drift apart faster than you'd think. Some defences:

- Store the markdown next to the diagnostic definitions, or in a dedicated `docs/errors/` directory in the same repo.
- Generate an index page listing every code with its `why` and severity. Easy to do programmatically from the `defineDiagnostics` call.
- **Validate in CI.** Walk the `codes` object, check that each entry has a corresponding page. Fail the build if a page is missing.
- When adding a new code, add the page in the same PR. The `/add-diagnostic` slash command (if you use the Claude plugin) is a good place to enforce this.

A minimal CI check:

```ts
import { existsSync } from 'node:fs'
import { diagnostics } from '../src/diagnostics'

const missing: string[] = []
for (const code of Object.keys(diagnostics)) {
  const path = `docs/content/errors/${code.toLowerCase()}.md`
  if (!existsSync(path))
    missing.push(code)
}
if (missing.length > 0) {
  console.error('Missing docs pages for:', missing)
  process.exit(1)
}
```

## Optimizing for AI agents

Agents are a real, growing audience for these pages. A few rules keep the content useful to them without harming the human experience:

- **Consistent heading hierarchy.** Always `##` for sections; don't skip levels.
- **Put the fix early.** The `How to fix it` section should be near the top of the page or one heading down from the title.
- **No collapsed sections, tabs, or accordion content** for critical material. Agents won't expand them.
- **Include the code in title and body text.** Keyword matching works on visible text.
- **Self-contained examples.** An agent should be able to suggest the fix from the page alone, without chasing more links.

Once the registry is in place, your diagnostics carry their own documentation everywhere they go. The terminal output points at a stable URL; that URL renders the same shape every time; and your library's error stories become navigable rather than hand-explained.
