## The problem

Library errors today are strings. Users scan them. Agents pattern-match them. Everyone loses context.

A typical thrown `Error` carries:

- A `message` that's often a paragraph of prose.
- A stack trace pointing into the library, not the user's code.
- No stable identifier. No suggested fix. No link to docs.

When something breaks, the user pastes the error into a search engine or an LLM and hopes. The library author has no way to point them at the right page, no way to evolve the wording without breaking that workflow, and no way to give an agent a structured handle to act on.

## What nostics is

A small, dependency-free library that turns each error and warning into a typed, serializable `Diagnostic`:

- A **stable code** (e.g. `MATH_E001`) that never changes once published.
- A **human `why`** that becomes the `Error.message`.
- An actionable **`fix`** string.
- A **`docs` URL** auto-derived from a base URL or function you control.
- Optional **`sources`** (`file:line:column` strings) when the JS stack doesn't reflect the user's source.
- Optional **`cause`** when re-raising from an original error.

You declare the catalog once. Every site that needs to raise a diagnostic gets a typed, cmd+clickable handle. Reporters fan it out to the console, a file, the network, or all three.

## What you get

- **Codes that survive refactors.** Reporters dispatch on `diagnostic.name`. Agents read it. Users search it.
- **A `Diagnostic` class that extends `Error`.** Throw it, catch it with `instanceof`, serialize it with `toJSON()`.
- **Pluggable reporters.** Console, file, HTTP, Vite dev-channel, or your own. Their options are inferred and type-checked at the call site.
- **Pluggable formatters.** Plain, ANSI, JSON. Or write your own.
- **A dev-time collector.** Vite plugin that tails browser diagnostics into a local log file, so agents and humans bound to the terminal can still see them.
- **A build-time stripper.** Marks `defineDiagnostics()` calls as pure and guards each call site with a `NODE_ENV` check, so diagnostics tree-shake out of production.

## When to use it

Reach for nostics when:

- You're authoring a library, framework, plugin, or CLI that surfaces errors and warnings to other developers.
- You expect to document those errors on a public page.
- You care about how agents and assistants react to your output, not just humans.

Skip it for:

- Internal one-shot scripts.
- Code that only ever throws to a single caller in the same process.
- Cases where a plain `throw new Error(...)` is the right tool.
